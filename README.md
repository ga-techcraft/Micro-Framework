# 開発過程メモ

## .envファイルについて
.envファイルを作成して、parse_ini_file() を使って連想配列として値を取得するコードを見たとき、「これ、phpファイルで連想配列を return すればよくない？」と思った。でも調べていくうちに、.envファイルを使うことには以下のようなメリットがあると気づいた。

【.env を使うメリット】
他の言語やツールでも扱える
→ .env は KEY=VALUE 形式のプレーンテキストなので、Node.js（process.env）、Python（os.getenv）、CI/CDツールなど、他の言語やシステムでも共通で読み込める。

セキュリティ面で安全に管理できる
→ .env を .gitignore に追加しておけば、パスワードやAPIキーなどがGitに上がらず漏洩リスクを防げる。

環境ごとに設定を分けられる
→ .env.local（開発用）と .env.production（本番用）などを用意して、getenv('APP_ENV') の値でどのファイルを読み込むか切り替えられる。

【PHPでの環境変数の扱い方】
・登録（書き込み）→ putenv("KEY=VALUE")
・取得（読み込み）→ getenv("KEY")

またステートレスクラスで、.envファイルからデータを取得するメソッドを定義することで、存在しないキーへのアクセス制御ができる。
さらに、例外をカスタマイズすることで、どこでエラーが起きたのかわかりやすくなる。

## mysqli_report()の設定
mysqli_report(MYSQLI_REPORT_ERROR | MYSQLI_REPORT_STRICT) を使うことで
MySQL接続時やクエリ実行時のエラーを例外としてスローしてくれるようになる。

## クラスは開発者の仕事の一部を肩代わりしてくれる
MySQLWrapperについては、以下のように。
- .env から設定を取得して代入する仕事をコンストラクタがやってくれる
- 接続エラーの報告や例外処理の設定もクラス内に隠せる
- 開発者は「インスタンス化するだけ」で接続できるようになる

Settingsクラスについて、以下のように。
- .env ファイルのキーが存在しないときに例外を投げることで
「設定ミスにすぐ気づけるようにする」安全設計になっている


## mysql -u {your_username} -p practice_db < Database/Examples/cars-setup.sql
シェルコマンドの < は入力リダイレクション演算子。これは、シェルに対して SQL コマンドを直接タイプインするのではなく、ファイルから入力を読み取るように指示できる。

## 手動スクリプト
- sql ファイルを作って mysql < ファイル名.sql で実行する方法は、初心者にも分かりやすく、直感的。
- Gitでバージョン管理できるので、大規模開発やチーム開発にも使える。
- スクリプトは簡単にカスタマイズ・再利用でき、環境ごとの切り替えにも便利。

## 既存DBにカラムや制約を追加するときの注意
- NULL可なら既存データ影響無しで追加できる。
- 外部キー制約、NOT NULL の追加は。まずはNULL可で追加。データを全件更新。その後にNOT NULL制約や外部キー制約を追加

## マイグレーションベース
- ファイル名にタイムスタンプやバージョン番号がつき、順番を自動判定
- マイグレーション履歴をDB内に記録（migrations テーブルなど）
- migrate コマンドで、未実行分だけ順番に適用
- rollback や refresh などでロールバック・再実行も自動化

## 文字列の中に変数を埋め込む方法
ダブルクオートで囲めば展開できるが、例えばテンプレートとなるコードの生成を作成したい場合、改行のためびに \n が必要になる。でも、ヒアドキュメントを使えば、自動で改行をしてくれて変数も埋め込めるから便利

htmlspecialchars() は、HTMLとして解釈されてしまう危険なテキストを、「ただの文字」として無効化する関数。

## 抽象クラス、インターフェース、トレイトの使い分け
- プロパティとメソッドの両方を共通化したい
→抽象クラス（abstract class）

- 共通のメソッドの“契約”だけを定義したい
→インターフェース（interface）

- 実装をそのまま使い回したい（コピペ的な再利用）
トレイト（trait）

で、抽象に依存するコードは、あとの変更やテストが楽になる。

## 実行モデルの違い
- PHP：リクエストごとにスクリプトを実行 → 終わったらプロセス終了（ステートレス）

- Node.js：1回アプリを起動 → イベントループでリクエストを処理し続ける（ステートフル）

PHP はリクエストごとにプロセスが立ち上がる。でも、その1回のリクエスト内で複数のDB操作があるとき、毎回接続し直すのは非効率。だから DatabaseManager が「すでに接続ある？」を確認して、同じ接続を使いまわす設計が必要。
→ この接続の使いまわしパターンがシングルトン。リクエストをまたいだシングルトンは使えないが、リクエスト内だけのシングルトンは可能。

## 設計目線
- クラスや関数に「1つの責任」を意識する
- 依存先を抽象にして疎結合を意識する

## データアクセス層
- 毎回SQLを書くのは大変＆エラーの元
- テーブルの1行と、**オブジェクト（モデル）**を対応させる
- データベース操作はDAOに任せる
- DAOはinterfaceで定義することで、MySQLだけでなく、SQLiteやPostgreSQLでも差し替え可能になる
